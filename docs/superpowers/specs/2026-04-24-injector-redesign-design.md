# Widget Injector Redesign — Design

**Date:** 2026-04-24
**Status:** Approved (pending user review of this doc)
**Owner:** Atul Nayyar

## 1. Goal

Rework the Studio **Injector** from a single-dropdown + single-example-payload form into a chip-based palette that exposes every widget and every variant at a glance. In the same change, make `widgetSchemas.js` the single source of truth for all example payloads, and refactor `mockBot.js` rules to read from it.

The Injector is developer-facing tooling. Its job is "pick a widget, pick a state, push it into the chat as a bot or a user message". Today that's three clicks, a hidden dropdown overflow, and no way to switch variants because only one example payload exists per widget.

## 2. Non-Goals

Explicitly out of scope:

- Widgets from the CSV spec that aren't in `src/chat/registry.js` yet (Date/Time Picker, Video Player, Comparison, Approval, Training Scenario, Location, Signature, Audio Player, Voice Recording, Embedded Webview, Payment/Earnings, Profile Card, Incentive/Leaderboard). When those are built, they add a `registry` entry plus a `widgetSchemas` entry and appear automatically.
- Type-to-search inside the Injector. Can be layered on later as a separate enhancement.
- Persisting the last-used widget/variant across page reloads.
- Unit / snapshot tests. Sandbox project — verification is manual.
- Changes to the mock-bot's text-trigger *regexes*. Only their `build()` bodies change.
- Any change to chat-side rendering (`ChatPane`, `MessageList`, widget components).

## 3. Problem

`src/studio/Injector.jsx` today:

- Uses `Select` from `@nexus/atoms`. With 20 widget types, the dropdown menu overflows the Studio panel viewport — options below the fold are not reachable on typical laptop heights.
- Reads `widgetSchemas[type].examplePayload` — one static payload per type. The concept of a "variant" (e.g. `confirmation` has `info` / `caution` / `danger` tones, `validated_input` has 7 input types) is encoded only as regex rules in `src/engine/mockBot.js`, not as selectable data.
- Example payloads in `widgetSchemas.js` and rule-body payloads in `mockBot.js` are duplicated by hand. Changes to a payload in one file silently drift from the other.

Requirements the redesign must satisfy:

1. All widgets visible without an overflowing dropdown.
2. All variants reachable in at most two clicks (widget → variant).
3. Single source of truth for example payloads — edit once, both trigger paths update.
4. Component-form implementation (no one-file blob) and existing token vocabulary (`var(--grey-*)`, `var(--brand-*)`, `var(--space-*)`). No raw hex or raw px for anything with a token.
5. Scalable: adding a new widget is one entry in `registry.js` + one entry in `widgetSchemas.js`. No Injector changes.

## 4. Data Model

### 4.1 `widgetSchemas.js` shape

Each entry moves from `{ label, examplePayload }` to `{ label, category, variants }`.

```js
export const widgetSchemas = {
  confirmation: {
    label: 'Confirmation Card',
    category: 'action',
    variants: [
      { id: 'info',    label: 'Info',    payload: () => ({ widget_id: makeId('confirm'), tone: 'info',    /* … */ }) },
      { id: 'caution', label: 'Caution', payload: () => ({ widget_id: makeId('confirm'), tone: 'caution', /* … */ }) },
      { id: 'danger',  label: 'Danger',  payload: () => ({ widget_id: makeId('confirm'), tone: 'danger',  /* … */ }) },
    ],
  },
  text: {
    label: 'Text message',
    category: 'engine',
    variants: [
      { id: 'default', label: 'Default', payload: () => ({ text: 'Hello from the injector.' }) },
    ],
  },
  // …
}
```

**Variant shape:** `{ id: string, label: string, payload: () => object }`.

- `payload` is a **function**, not an object. Invoked on every selection so ids generated by `makeId()` are fresh each time.
- Widgets with no meaningful variants still carry a single `{ id: 'default', label: 'Default' }` entry. The UI then has no special cases.
- `category ∈ 'action' | 'input' | 'display' | 'advanced' | 'engine'`. Drives grouping in the palette, nothing else.

### 4.2 Public helper

```js
export function getVariantPayload(type, variantId) {
  const variant = widgetSchemas[type]?.variants.find((v) => v.id === variantId)
  return variant?.payload() ?? {}
}
```

Used by `mockBot.js` rules. The Injector uses the schemas directly (it needs labels too).

### 4.3 `examplePayload` removal

The current `examplePayload` field is read only by `Injector.jsx`. After the migration it no longer exists — there is no backwards-compatibility alias. Each widget's former `examplePayload` becomes `variants[0].payload()`.

### 4.4 Shared data: `richJobs()`

`mockBot.js` currently exports a `richJobs()` helper used by two rules (`job_card` carousel, `carousel` of `job_card`s). It moves into `widgetSchemas.js` alongside the `job_card` and `carousel` entries, since those are now the canonical home for these payloads. `mockBot.js` no longer needs to export it.

20 widget types currently in `registry.js`, yielding **42 variants** total. Categories follow the CSV spec; `text` / `widget_response` are engine utilities (not in the CSV) and grouped separately.

**Seed payloads:** harvested from the existing `build()` bodies in `mockBot.js` — these are the payloads the mock bot already produces for the equivalent text triggers. Three variants are authored fresh (no equivalent in `mockBot.js` today):

- `instruction_card#warn` — same shape as the existing `info` variant with `tone: 'warn'` and warn-toned copy.
- `instruction_card#success` — same shape as the existing `info` variant with `tone: 'success'` and success-toned copy.

These three will also get matching regex rules in `mockBot.js` so the "single source of truth" property holds — every variant reachable via chip is also reachable via a text trigger.

## 6. Component Structure

Five files in `src/studio/`. Matches the existing `BrandPicker` / `Controls` / `ViewportToggle` pattern (one JSX + one `.module.scss` per component).

```
src/studio/
├─ Injector.jsx              ← orchestrator: state + wiring
├─ injector.module.scss      ← shell layout only
├─ WidgetPalette.jsx         ← grouped radiogroup of widget chips
├─ widgetPalette.module.scss
├─ VariantRow.jsx            ← radiogroup of variant chips (null if 1 variant)
├─ variantRow.module.scss
├─ PayloadEditor.jsx         ← labelled textarea + inline error slot
└─ payloadEditor.module.scss
```

### 6.1 `Injector.jsx` (orchestrator)

State (`useState`):

- `widgetType: string` — initial `Object.keys(widgetSchemas)[0]`
- `variantId: string` — initial `widgetSchemas[widgetType].variants[0].id`
- `payloadText: string` — initial stringified result of the initial variant's `payload()`
- `error: string | null`

Derived:

- `useMemo` over `Object.entries(widgetSchemas)` filtered by "also exists in `registry`", grouped by `category` in fixed order (`action`, `input`, `display`, `advanced`, `engine`).
- `useMemo` for the active widget's variants.

Props passed down:

- To `WidgetPalette`: `{ groups, selected: widgetType, onSelect }`
- To `VariantRow`: `{ variants, selected: variantId, onSelect }`
- To `PayloadEditor`: `{ value: payloadText, error, onChange }`
- Inject buttons: `onInjectBot`, `onInjectUser` (call `bot.injectBotMessage` / `bot.injectUserWidgetResponse` after `JSON.parse`).

### 6.2 `WidgetPalette.jsx`

Renders categories as sections. Each section has a small label (`font-size: 11px`, `color: var(--grey-50)`, uppercase) and a `flex-wrap` row of chips inside it.

```jsx
<div className={styles.palette}>
  {groups.map(({ category, label, widgets }) => (
    <section key={category}>
      <h3 className={styles.groupLabel}>{label}</h3>
      <div role="radiogroup" aria-label={`${label} widgets`} className={styles.row}>
        {widgets.map((w) => (
          <button
            key={w.type}
            type="button"
            role="radio"
            aria-checked={w.type === selected}
            className={cx(styles.chip, w.type === selected && styles.active)}
            onClick={() => onSelect(w.type)}
          >
            {w.label}
          </button>
        ))}
      </div>
    </section>
  ))}
</div>
```

Category display labels:
- `action` → "Action & Interaction"
- `input` → "Input & Data Collection"
- `display` → "Display & Information"
- `advanced` → "Embedded & Advanced"
- `engine` → "Engine"

### 6.3 `VariantRow.jsx`

Returns `null` when `variants.length < 2`. Otherwise: a single `role="radiogroup"` wrapping variant chips using the same chip visual spec as `WidgetPalette`. The chip class rules are duplicated across `widgetPalette.module.scss` and `variantRow.module.scss` — identical ~15 lines of CSS in each, no shared file. Duplication is cheap at this scale and keeps each component self-contained.

### 6.4 `PayloadEditor.jsx`

- Label "Payload (JSON)".
- Always-visible `Textarea` from `@nexus/atoms`, `rows={8}`.
- Below the textarea: an error slot with `role="alert"`. Shows `Invalid JSON: <message>` when `error` is non-null. Otherwise collapsed (no empty div height).

### 6.5 Chip visual spec

All tokens from Nexus. No raw hex, no raw px where a token exists.

| State | Tokens |
|---|---|
| Idle | `background: var(--grey-05)` · `border: 1px solid var(--grey-20)` · `color: var(--grey-80)` |
| Hover | `background: var(--grey-10)` |
| Active | `background: var(--brand-600)` · `color: #fff` · `border-color: transparent` |
| Focus-visible | `outline: 2px solid var(--brand-400)` · `outline-offset: 2px` |

Sizing: `padding: var(--space-2) var(--space-3)` · `min-height: 28px` · `border-radius: var(--radius-sm)` · `font-size: 12px` · `font-family: var(--sans)`. The one literal in the chip CSS is `#fff` for active-chip text, which is the inverse-on-brand pattern the existing codebase uses (not a "design hex").

```
┌─ Widget ────────────────────────────────────────────┐
│  ACTION & INTERACTION                                │
│   [Quick reply] [Confirmation] [Checklist] [Shifts]  │
│  INPUT & DATA COLLECTION                             │
│   [MCQ] [File upload] [Image capture] [Form] …       │
│  DISPLAY & INFORMATION                               │
│   [Progress] [Score] [Job card] [Carousel] …         │
│  EMBEDDED & ADVANCED                                 │
│   [QC review] [Evidence review]                      │
│  ENGINE                                              │
│   [Text] [Widget response]                           │
├─ Variant ───────────────────────────────────────────┤
│  [Info]  [Caution]  [Danger]           ← hidden if schema has only 1 variant
├─ Payload (JSON) ────────────────────────────────────┤
│  { … editable textarea, always visible … }           │
│  ⚠ Invalid JSON: …                                   │
├─────────────────────────────────────────────────────┤
│  [Inject as bot]    [Inject as user]                 │
└─────────────────────────────────────────────────────┘
```

Chips wrap. At the narrowest Studio width (~260px) each row of chips may become 1–2 chips wide — that's acceptable. No horizontal scroll, no overflow popovers.

## 8. Interaction Contract

| Trigger | Behaviour |
|---|---|
| Mount | First widget in the first category is selected. Its first variant is selected. Textarea shows `variant.payload()` pretty-printed. |
| Click widget chip | If different widget → select it, reset variant to its first, regenerate payload. Same widget → no-op. |
| Click variant chip | Regenerate payload by invoking `variant.payload()` (fresh ids). |
| Edit textarea | Kept verbatim. Any edits are **discarded** when the widget or variant changes — no confirm dialog. |
| Inject as bot | `JSON.parse(payloadText)`. If valid → `bot.injectBotMessage({ type: widgetType, payload: parsed })`. If invalid → show error, no message injected. |
| Inject as user | Same as bot, but `bot.injectUserWidgetResponse({ type: widgetType, payload: parsed })`. |
| Parse error | Inline red error under the textarea, `role="alert"`. Buttons stay enabled so the user can type-and-retry. |
| `bot.reset` | No effect on Injector state. Selections persist until the tab reloads. |

Keyboard:

- `Tab` moves between the three radiogroups (widget palette, variant row, textarea) and the two action buttons.
- Inside a radiogroup: `←/→` or `↑/↓` cycles chips and updates selection; `Space` / `Enter` also activate the focused chip.
- `aria-checked` on the active chip drives screen-reader state.

## 9. Migration Plan

### 9.1 Extract `makeId` to avoid circular imports

`makeId` currently lives in `mockBot.js`. After the refactor, `widgetSchemas.js` needs it (variant payloads call it) and `mockBot.js` needs `getVariantPayload` from `widgetSchemas.js` — a circular import. Resolve by extracting `makeId` into a new tiny module:

```
src/engine/ids.js
```

```js
import { v4 as uuid } from 'uuid'
export function makeId(prefix = 'w') {
  return `${prefix}-${uuid().slice(0, 8)}`
}
```

Both `widgetSchemas.js` and `mockBot.js` import from there. `mockBot.js` no longer exports `makeId`.

### 9.2 `widgetSchemas.js`

1. Import `makeId` from `./ids.js`.
2. Move `richJobs()` from `mockBot.js` into `widgetSchemas.js`.
3. Replace every `{ label, examplePayload }` entry with `{ label, category, variants: [...] }`.
4. Seed each variant's `payload` function from the corresponding `build()` body in `mockBot.js`. Three variants are authored fresh per §5: `qc_evidence_review#worker`, `instruction_card#warn`, `instruction_card#success`.
5. Export `getVariantPayload(type, variantId)`.

### 9.3 `mockBot.js`

1. Import `makeId` from `./ids.js` (for the `defaultEcho` path, which still needs it).
2. `import { getVariantPayload } from './widgetSchemas.js'` and remove the inline `richJobs` definition.
3. Each `registerRule` body becomes `build: () => ({ type, payload: getVariantPayload(type, variantId) })`.
4. Add three new `registerRule` entries for `qc_evidence_review#worker`, `instruction_card#warn`, `instruction_card#success` so all chip-reachable variants are also text-trigger-reachable.
5. Leave existing regex matches and rule ordering unchanged — the goal is zero behaviour change for text-typed triggers that exist today.
6. Delete the `_UNUSED_OLD_JOBS` dead block at the bottom of the file.

### 9.4 `Injector.jsx`

Full rewrite. Replaces the `Select` import entirely. Props unchanged (`{ bot }`).

### 9.5 `StudioPanel.jsx`

Unchanged — still renders `<Injector bot={bot} />` inside the existing section.

## 10. Edge Cases

- **Schema / registry mismatch:** any `widgetSchemas` key missing from `registry` is filtered out of the palette at render time with a single `console.warn`.
- **0-variant schema:** guarded with `variants?.[0] ?? null` plus `console.warn`. Invariant is that every schema has ≥1 variant; the guard exists so a developer typo doesn't crash the Studio.
- **`payload()` throws:** wrap the invocation in a try/catch; on failure, set textarea to `{}` and surface the message in the error slot.
- **Long chip labels:** wrap to 2 lines inside the chip. No ellipsis.
- **Narrow panel:** chips wrap via `flex-wrap: wrap`. No additional media queries.

## 11. Verification

Manual, same as the rest of the Studio:

1. Each of the 20 widget chips is reachable; clicking it flips to its first variant and the payload updates.
2. Widgets with multiple variants show a variant row; widgets with one variant do not.
3. "Inject as bot" and "Inject as user" both work for every variant.
4. Editing the textarea then clicking inject uses the edited JSON.
5. Invalid JSON surfaces an inline error and does not inject.
6. Typing each existing mock-bot text trigger (e.g. `show caution confirm`, `otp`, `stars`) produces an identical widget to clicking the matching chip. This is the single-source-of-truth check.
7. `console.warn` count is zero on a healthy session.
